## 1. 文档目的与读者

本文档汇总「求职顾问」应用的产品与技术设计，统一**项目术语**、**数据模型与枚举**、**Agent 流程**与**对外接口**，作为研发与联调的依据。实现阶段：为加快落地，**业务逻辑中的规则与抽取优先由大语言模型承担**；基础设施通过 Docker Compose 拉起，并附带各组件的管理/控制台。

---

## 2. 项目概述

求职顾问是一个面向求职场景的对话式应用：用户通过自然语言与系统交互，系统完成**任务分拣、子任务规划、工具执行、结果审核与回复生成**，并结合**简历、职位、用户偏好事实**等数据提供个性化建议。短期对话上下文存 Redis；结构化与向量检索分别使用 Elasticsearch 与 **ChromaDB**；主数据与关系使用 MongoDB。

## 3. 技术架构

| 层级 | 技术选型 | 说明 |
| --- | --- | --- |
| 前端 | React + TypeScript + Vite | 通过 REST API 与后端交互；聊天流式接口使用 **JSON Lines（JSONL）**，前端用 `ReadableStream` 按行解析 |
| 后端运行时 | Python，依赖由 **uv** 管理 | `pyproject.toml` + `uv.lock` 固定版本 |
| Web 框架 | FastAPI | REST + 流式响应 |
| Agent / LLM | LangChain、LangGraph | 编排 Triage / Plan / Review / Executor |
| 数据校验 | Pydantic | 请求体、配置与部分领域模型 |
| 主数据库 | MongoDB | 任务、职位、用户职位、简历等 |

| 全文 / 结构化检索 | Elasticsearch（ES） | 事实存储与检索 |
| 向量检索 | ChromaDB | 事实的向量副本，与 ES 侧**语义一致的数据** |
| 容器编排 | docker-compose.yml | 部署 MongoDB、Redis、ES、ChromaDB **及各自控制台/管理 UI** |

**控制台约定（Compose 中需包含的典型组件，名称以实际镜像为准）：**

| 服务 | 常见控制台 / 管理端 | 用途 |
| --- | --- | --- |

| Redis | RedisInsight 或 redis-commander | 查看 key、TTL、内容 |
| Elasticsearch | Kibana 或 Elasticvue | Dev Tools、索引与查询 |
| ChromaDB | HTTP API（可选 Admin） | 向量集合与 metadata 过滤调试 |

## 5. 项目术语表

| 术语 | 英文 / 别名 | 定义 |
| --- | --- | --- |
| 主任务 | Task | 由用户一次「对话入口」发起的顶层记录，对应表 `tasks`，生命周期可被定时任务关闭 |
| 子任务 | Sub-task | 规划层在一次主任务下追加的可执行单元，对应表 `sub_tasks` |
| 分拣 | Triage | 判断当前用户输入是否归属「最近」主任务；不相关则新建主任务 |
| 规划 | Plan | 结合混合检索得到的事实与子任务类型，由 LLM 决定追加哪些子任务 |
| 审核 | Review | 将子任务从创建/等待推进为成功/失败，并处理工具产出与缓存 |
| 执行 | Executor | 聚合 Redis 历史、事实、工具缓存与当前输入，调用 LLM 生成面向用户的回复 |
| 事实 | Fact | 从对话中抽取的、与求职相关的结构化断言，写入 ES 与 ChromaDB，可合并/更新/作废 |
| 混合检索 | Hybrid retrieval | 同时使用 ES（关键词/过滤等）与 ChromaDB（向量）检索事实，结果进入 Planner / Executor 上下文 |
| 工具缓存 | Tool result cache | Review 阶段对「可返回信息的工具」结果做缓存，供主任务维度读取；Executor 优先读缓存 |
| JSONL 流 | NDJSON stream | `/chat/stream` 每行一个 JSON 对象，便于前端流式解析 |
| 业务编号 | Business ID / Job biz id | 职位在业务侧的稳定编号，用于 `searchJobs` 等；优先引导用户提供 |

## 6. MongoDB 数据库设计

### 6.1 集合：`tasks`（主任务）

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | ObjectId / string | 主键 |
| `user_id` | string | 用户标识 |
| `user_input` | string | 发起任务时的用户输入（可摘要或全文，实现时约定） |
| `status` | string | 见下表 **TaskStatus** |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |

**TaskStatus（主任务状态）**

| 枚举值 | 说明 |
| --- | --- |
| `active` | 进行中，可接收新对话与子任务 |
| `closed` | 已关闭（超时规则或业务关闭） |
| `archived` | 归档（可选，用于历史查询优化） |

**关闭规则（与术语一致）：** 定时任务将主任务标为 `closed`，当 **最后一条子任务创建时间距今 ≥ 30 分钟**，或 **主任务创建时间距今 ≥ 2 小时**（实现时对「或」的优先级以产品为准，文档建议：**满足任一即关闭**）。

---

### 6.2 集合：`sub_tasks`（子任务）

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | ObjectId / string | 主键 |
| `task_id` | string | 外键 → `tasks.id` |
| `task_type` | string | **SubTaskType** |
| `status` | string | **SubTaskStatus** |
| `params` | object | 工具名、槽位、事实确认参数等 JSON |
| `tool_name` | string | 可选，便于索引与去重（如与 `params` 冗余） |
| `result_cache_key` | string | 可选，指向工具结果缓存 |
| `error_message` | string | 可选，失败原因 |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |

**SubTaskType（子任务类型）**

| 枚举值 | 说明 |
| --- | --- |
| `tool_call` | 调用具体工具（如 searchJobs） |
| `tool_slot_clarify` | 槽位不清时的追问；追问文案由**硬规则**生成，非完全依赖 LLM 自由发挥 |
| `fact_confirm` | 对抽取事实的确认（展示/确认/纠正） |

**SubTaskStatus（子任务状态）**

| 枚举值 | 说明 |
| --- | --- |
| `created` | 已创建，待调度 |
| `waiting` | 等待前置条件（如用户补充槽位） |
| `running` | 执行中 |
| `success` | 成功 |
| `failed` | 失败 |

**去重提示：** Plan 追加 `tool_call` 时，若与已有子任务**相同工具 + 相同参数**（实现上可对 `params` 规范化后哈希比较），应**提示用户**避免重复执行（具体文案产品定）。

## 7. Redis 设计

| 项 | 说明 |
| --- | --- |
| 逻辑名 | `user_history_dialogue` |
| 维度 | 以 `task_id` 维度记录用户与智能体短期对话 |
| Key 模式 | `user:{user_id}_{task_id}`（与需求一致；实现时可改为 `user:{user_id}:task:{task_id}` 以降低歧义） |
| Value | 消息列表或序列化消息（实现定：如 Redis List 存 JSON 字符串） |
| TTL | **2 小时** |

**用途：** Executor 拉取当前任务历史；Triage 可读取**最近一条 task** 在 Redis 中的对话，判断当前输入是否仍属该主任务。

## 8. Elasticsearch 与 ChromaDB（事实存储）

### 8.1 事实文档模型（ES 与 ChromaDB 语义一致）

示例（与需求对齐）：

```json
{
  "user_id": "usr_001",
  "category": "constraint",
  "subject": "user",
  "predicate": "pref_location",
  "value": "上海或远程",
  "content": "用户目前只考虑上海或远程",
  "entities": {"location": ["上海"], "work_mode": ["onsite", "remote"]},
  "confidence": 0.88,
  "status": "active",
  "created_at": "2025-03-21T10:00:00Z"
}
```

### 8.2 事实相关枚举

**FactCategory（事实类别）**

| 枚举值 | 说明 |
| --- | --- |
| `constraint` | 硬性约束（地点、出差等） |
| `preference` | 偏好（行业、公司规模等） |
| `profile` | 履历类事实（年限、技能等级等） |
| `behavior` | 行为类（已投递某公司） |

**FactSubject（主语类型）**

| 枚举值 | 说明 |
| --- | --- |
| `user` | 用户本人 |

**FactStatus（事实生命周期）**

| 枚举值 | 说明 |
| --- | --- |
| `active` | 当前有效 |
| `superseded` | 被新事实合并/替代 |
| `revoked` | 用户明确否定或删除 |

**Predicate（谓词，与求职相关，建议白名单）**

| predicate | 含义 |
| --- | --- |
| `pref_location` | 地点偏好 |
| `pref_work_mode` | 工作模式偏好（现场/远程/混合） |
| `avoid_business_trip` | 是否避免出差 |
| `years_experience` | 工作年限 |
| `skill_level` | 技能水平（可按技能名扩展 entities） |
| `onboard_date` | 可到岗时间 |
| `industry_pref` | 行业偏好 |
| `company_size_pref` | 公司规模偏好（需求中 `company_size_of` 建议统一为此命名） |
| `applied_company` | 已投递/已申请的公司 |

抽取由 LLM 完成，范围限制在**求职相关**；本地提供 **function_call** 供模型触发 **合并、更新、删除**；查询接口支持 **`user_id` + `predicate`** 获取当前有效事实集。

**ChromaDB：** 与 ES 存**同一条事实**的向量（如对 `content` 或 `value` 嵌入），`document id` 与 ES 的 `fact_id` 对齐；用于语义检索。Plan / Executor 的混合检索结果需**进入上下文**再交给下游。

## 9. Agent 设计

### 9.1 总览

| 阶段 | 职责 |
| --- | --- |
| Triage | 判断是否关联「最近」主任务；关联则传递 `task_id`，否则新建任务 |
| Plan | 混合检索事实 → 上下文；LLM 追加子任务（含去重提示）；`tool_slot_clarify` 追问由硬规则生成 |
| Review | 子任务状态流转；区分**有信息产出**与**仅 DB 变更**的工具；有信息则写入缓存供主任务读取；读工具结果时**优先缓存** |
| Executor | Redis 历史 + 工具缓存 + 混合检索事实 + 当前输入 → LLM 生成回复；回复过程中**异步触发事实抽取** |

### 9.2 Triage 细则

- 候选主任务：从 `tasks` 中取 **近 30 分钟内创建** 的记录（与「主任务关闭」时间尺度区分：此处为**相关性判断窗口**）。
- 结合**最近一条 task** 在 Redis 中的对话，判断当前用户输入是否仍属该任务语义。
- 若不相关：创建新 `tasks` 记录。

### 9.3 主任务定时关闭

- 周期任务扫描：若 **最后子任务 `created_at` 距今 ≥ 30 分钟**，**或** **主任务 `created_at` 距今 ≥ 2 小时**，则将 `tasks.status` 置为 `closed`（若采用「无子任务」边界条件，实现时单独约定）。

### 9.4 并发与一致性

文档约定：**不引入分布式锁**，以简化实现；接受极端并发下的短暂不一致（如重复子任务需靠上层提示与幂等设计缓解）。

## 10. 工具（Tools）定义

| 工具名 | 作用 | 是否「有信息返回」供缓存 |
| --- | --- | --- |
| `searchJobs` | 按**业务编号列表**或 `title` 查职位；**优先业务编号** | 是 |
| `searchResume` | 按 `user_id` 查唯一简历 | 是 |
| `searchCompany` | 查公司信息，**可触发联网搜索** | 是 |
| `updateJobs` | 更新用户职位状态、关注程度等 | 否（仅 DB 变更，标注为无结构化「查询结果」缓存） |
| `updateResume` | 更新简历 | 否 |
| `notifyUser` | 标记某操作完成，配合前端提示 | 否（走通知通道） |

Review 层必须维护**工具元数据**：`returns_fetchable_payload: bool`，用于决定是否写入工具结果缓存。

## 12. 设计理解与待实现时确认点

| 序号 | 主题 | 理解摘要 | 待确认 |
| --- | --- | --- | --- |
| 1 | 模型驱动 | 规则判断、事实抽取、规划等优先 LLM，本地以 function_call 约束边界 | 模型供应商、密钥与降级策略 |
| 2 | Triage 时间窗 | 文档将「30 分钟内 tasks」用于相关性；「30 分钟无子任务 / 2 小时」用于关闭 | 两处置的 30 分钟是否应统一为同一常量 |
| 3 | Redis key | `user:{userid_taskid}` 需避免 `user_id` 含下划线时歧义 | 是否改为分层 key |
| 4 | 工具缓存 | 无锁下缓存键建议含 `task_id` + `sub_task_id` 或工具调用 id | 缓存 TTL 与淘汰策略 |
| 5 | 通知 | `notifyUser` 与 `notify_api` 的实时通道 | SSE / WebSocket / 短轮询 |
